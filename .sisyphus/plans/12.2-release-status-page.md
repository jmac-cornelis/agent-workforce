## 4. Jira Deficiencies Found

### 🔴 Critical

| # | Issue | Impact | Evidence |
|---|-------|--------|----------|
| C1 | **319 CN6000 tickets have no fixVersion** | Cannot track CN6000 12.2 status — work is invisible | `project=STL AND "Product Family"=CN6000 AND fixVersion is EMPTY AND status not in (Closed,Done)` returns 319 |
| C2 | **582 open bugs have no fixVersion** | 12.2 bug count is artificially low; unknown number of these should be in 12.2 | `project=STL AND issuetype=Bug AND fixVersion is EMPTY AND status not in (Closed,Done)` returns 582 |
| C3 | **12.2.0.x release date is 2025-10-31** | 5 months stale — velocity/readiness estimates are meaningless without a real target | Jira version `12.2.0.x` shows `releaseDate: 2025-10-31` |
| C4 | **SW Release tracker tickets have wrong fixVersions** | Release anchors are mislinked, breaking traceability | STL-75304 ("12.2.1") → fixVersion `12.1.1.x`; STL-75299 ("12.3.0") → fixVersion `12.1.0.x` |

### 🟡 Moderate

| # | Issue | Impact |
|---|-------|--------|
| M1 | No milestone/gate tracking | No alpha/beta/RC/GA visibility; can't answer "where are we in the release cycle" |
| M2 | 15+ obsolete components in project | Pollutes component picker, risk of mis-tagging |
| M3 | No cross-project ASIC visibility | CN5AW/CYRAW blockers invisible from STL queries |
| M4 | `Found In Build` / `Fixed In Build` may not be implemented | Can't answer "which build has this fix" for verification |
| M5 | 12.2 is 98% CN5000 — CN6000 is a sidecar at best | Planning ambiguity: is 12.2 a joint release or not? |

## 5. Remediation Plan

### Phase 1: Immediate (This Week)

**P1-A. Triage CN6000 fixVersions**
- Query: `project=STL AND "Product Family"=CN6000 AND fixVersion is EMPTY AND status not in (Closed,Done)`
- Decision needed per ticket: assign to 12.2.0.x, 12.3.0.x, or mark as backlog/unplanned
- Owner: Product management
- This is the single biggest gap — without it, CN6000 tracking is fiction

**P1-B. Update 12.2.0.x release date**
- Set a realistic target date in the Jira version
- This unblocks velocity/readiness calculations

**P1-C. Fix SW Release tracker fixVersions**
- STL-75304: Change fixVersion from `12.1.1.x` to `12.2.0.x` (or create `12.2.1.x`)
- STL-75299: Change fixVersion from `12.1.0.x` to `12.3.0.x`

**P1-D. Triage unversioned open bugs**
- Query: `project=STL AND issuetype=Bug AND fixVersion is EMPTY AND status not in (Closed,Done)`
- Subset to review: sort by priority (P0/P1 first), then by `updated DESC`
- Minimum: assign fixVersion to any P0/P1 bugs that should be in 12.2

### Phase 2: Short-term (This Sprint)

**P2-A. Establish release gate labels**
- Options (pick one):
  - Label convention: `gate:dev`, `gate:integration`, `gate:rc`, `gate:ga`
  - Formalize existing: `sw_debugging`, `sw_critical_debugging` + add `sw_verification`, `sw_rc`
  - Custom field: "Release Phase" dropdown
- Apply to all 12.2.0.x tickets

**P2-B. Create saved JQL filters**
```
12.2 All:           project=STL AND fixVersion="12.2.0.x"
12.2 P0/P1:         project=STL AND fixVersion="12.2.0.x" AND issuetype=Bug AND priority in ("P0-Stopper","P1-Critical") AND status not in (Closed,Done)
12.2 CN5000 Bugs:   project=STL AND fixVersion="12.2.0.x" AND issuetype=Bug AND "Product Family"=CN5000
12.2 CN6000 Bugs:   project=STL AND fixVersion="12.2.0.x" AND issuetype=Bug AND "Product Family"=CN6000
12.2 Features:      project=STL AND fixVersion="12.2.0.x" AND issuetype=Epic
12.2 Unversioned:   project=STL AND "Product Family" in (CN5000,CN6000) AND fixVersion is EMPTY AND status not in (Closed,Done)
12.2 No Assignee:   project=STL AND fixVersion="12.2.0.x" AND assignee is EMPTY
```

**P2-C. Archive obsolete components**
- Archive 15+ "Obsolete -" prefixed components (preserves history, removes from picker)

**P2-D. Build the Confluence page**
- Use structure from Section 3 above
- Embed JQL macros from saved filters (P2-B)
- Add manual sections for executive summary and gate status

### Phase 3: Medium-term (Next 2-4 Weeks)

**P3-A. Cross-project ASIC linking**
- Create blocking links from CN5AW/CYRAW tickets to dependent STL tickets
- Makes ASIC blockers visible in release health reports

**P3-B. Automate status page refresh**


**P3-C. Implement `Found In Build` / `Fixed In Build` governance**
- Per the version-field-governance-proposal.md (dated 2026-03-13, status: Draft)

**P3-D. Add trend visualization**
- Weekly bug open/close trend chart (matplotlib or Confluence chart macro)

## 6. Strategic Decisions Needed

These are product management decisions, not engineering tasks:

1. **Is 12.2 a joint CN5000+CN6000 release or CN5000-primary?**
   - Currently 98% CN5000 bugs, only 31 CN6000 tickets versioned to 12.2
   - 319 CN6000 tickets unversioned — are these 12.2 or 12.3?
   - The answer determines whether the status page is one view or two

2. **What is the real 12.2 release date?**
   - Current Jira date: 2025-10-31 (stale)
   - Without this, velocity and readiness estimates are meaningless

3. **What are the release gate criteria?**
   - When is "feature complete"?
   - What P0/P1 count triggers release hold?
   - What verification criteria must pass?

4. **How should the 582 unversioned open bugs be handled?**
   - Bulk triage session? Component-owner triage? Auto-assign to backlog?

## 8. Recommended Build Approach

### Option A: Manual Confluence Page (Fastest)
- Build the page structure manually in Confluence
- Embed saved JQL filters as Jira Issues macros
- Update executive summary manually weekly
- **Effort: 1-2 hours to set up, 30 min/week to maintain**

### Option B: Semi-Automated (Recommended)
- Build the Confluence page structure manually
- Use `gantt-release-monitor` for weekly data refresh (CN5000 and CN6000 separately)
- Copy key metrics from monitor output into Confluence
- JQL macros stay live automatically
- **Effort: 2-3 hours setup, 15 min/week to maintain**

### Option C: Fully Automated (Most Investment)
- Build a new workflow or extend `gantt-release-monitor` for cross-product comparison
- Add Confluence API integration to auto-publish
- Add charting (matplotlib) for trend visualization
- Schedule via poller
- **Effort: 1-2 weeks engineering, near-zero maintenance**

**Recommendation: Start with Option B.** It delivers 80% of the value immediately. Invest in Option C only if the page proves valuable and the manual refresh becomes burdensome.

```sql
  ORDER BY priority ASC, updated DESC

project = STL AND fixVersion = "12.2.0.x" AND issuetype = Bug

project = STL AND fixVersion in ("12.1.0.2.x", "12.1.2.x", "12.2.0.x", "12.3.0.x")
```
