# 12.2 Release Cockpit — Unified Recommendations

**Date:** 2026-03-25
**Author:** John Macdonald
**Status:** Draft
**Inputs:** Live Jira survey (2026-03-25) + STL_12_2_RELEASE_COCKPIT_BLUEPRINT.md
**Scope:** Trustworthy release cockpit for `12.2.0.x` spanning CN5000 and CN6000

---

## 0. Why This Document

Two independent analyses of the `12.2.0.x` release produced overlapping but distinct findings:

- **Analysis A** (live Jira survey): Data-heavy. Found 267 tickets via broad `fixVersion` query, quantified the CN6000 gap (319 unversioned), found wrong fixVersions on SW Release trackers, cataloged 582 unversioned open bugs, inventoried existing automation tooling.
- **Analysis B** (cockpit blueprint): Model-heavy. Found 100 tickets via scoped query (fixVersion + labels/Product Family), identified hierarchy defects (8 orphan stories, 3 multi-release feature tickets, 2 stale epics), defined canonical Jira data model, operating rules, and ownership model.

The count discrepancy (267 vs 100) itself demonstrates the core problem: **scope is modeled inconsistently**. Depending on whether you query by `fixVersion` alone or require `fixVersion` AND `Product Family` / `labels`, you get a different answer. Both analyses agree the data model must be fixed before the page can be trusted.

This document merges both into a single recommendation set.

---

## 1. Guiding Principle

> **Fix the model first. The page is just a rendering layer over trustworthy data.**

A status page built on inconsistent Jira data creates a false sense of control. The right sequence is:

1. Establish a canonical Jira model (fields, hierarchy, rules)
2. Repair the data to match the model
3. Create saved filters against the clean data
4. Build the Confluence page from those filters
5. Automate refresh where Jira macros alone are insufficient

## 2. Canonical Jira Model

### Source-of-truth fields

| Field | Role | Authority |
|-------|------|-----------|
| `fixVersion` | Release dimension | Primary — every in-scope ticket must have exactly one (features) or intentional multiples (bug backports only) |
| `Product Family` (`customfield_28382`) | Scope dimension | Primary — authoritative for CN5000 / CN6000 / shared classification |
| `parent` | Hierarchy dimension | Primary — every Story/Task must roll up to an Epic |
| `status` | Workflow state | Primary |
| `priority` | Urgency | Primary |
| `labels` | Convenience/tagging | **Secondary** — not a reporting field. Do not use labels as a primary filter for release scope |

**Key rule:** `Product Family` is the canonical scope field. Labels may duplicate this for convenience, but filters and reports must use `Product Family`.

### Issue-type roles

| Type | Role in release |
|------|----------------|
| `SW Release` | Release-operations umbrella (one per release) |
| `Initiative` | Family-level delivery root (one per family per release) |
| `Epic` | Meaningful deliverable bucket |
| `Story` / `Task` / `Technical task` | Execution work under an Epic |
| `Bug` | Release-readiness work — tracked separately from feature completion |
| `Sub-task` | Child of execution work |

### Recommended hierarchy

```
SW Release: 12.2.0.x (STL-75294 or replacement)
├── Initiative: 12.2.0.x CN5000 Feature Delivery (new or existing)
│   ├── Epic: SLES16.0 DevOps (STL-76008)
│   ├── Epic: JKR cleanup board revisions (STL-75423)
│   ├── Epic: BMC IPoIB (STL-69162)
│   ├── Epic: MYR Linux Driver ipoib (STL-68268)
│   └── ...
├── Initiative: 12.2.0.x CN6000 Feature Delivery (NEW — does not exist today)
│   ├── Epic: Merge BTS/Verbs to CN6000/CN5000 (STL-76300)
│   ├── Epic: CN6000 OPA/HFIsvc enabling (STL-75118)
│   ├── Epic: CYR General Cport Updates (STL-70567)
│   └── ...
└── Bugs: linked to features when blocking, but NOT children of epics
```

### Critical modeling rule

**Feature progress must not be inferred from bug counts.**

- Feature delivery = non-Bug issues (Stories, Tasks, Epics)
- Bug readiness = Bug issues only
- Shared/cross-family work shown separately, not silently duplicated in totals
- Bugs link to blocked feature work via `blocks` links when relevant

## 6. Cockpit Page Structure

A single Confluence page with live Jira-backed sections and generated analysis where Jira macros alone are insufficient.

### Section 1: Header

- Release name: `12.2.0.x`
- Covered families: CN5000, CN6000
- Refresh timestamp
- Release owner
- Overall health: `On Track` / `At Risk` / `Off Track`
- Top 3 release risks
- Top 3 decisions needed this week

*Source: Generated summary + Jira release metadata + manual owner field*

### Section 2: Scope & Definitions

Static section that prevents confusion about what the page means:

- **Feature Delivery** = non-Bug issues (Stories, Tasks, Epics)
- **Bug Readiness** = Bug issues only
- **Shared Work** = tickets spanning more than one product family
- **Status mapping**: Done / In Progress / Remaining / Blocked
- **Scope fields**: `fixVersion` (release) + `Product Family` (family)
- **What is NOT shown**: Tickets without `fixVersion`, tickets in other releases

*Source: Static template text, maintained in page*

### Section 3: Executive Summary

Short narrative for management — no tables required:

- Release summary paragraph
- CN5000 status summary
- CN6000 status summary
- Bug-readiness summary
- Data-quality summary (one sentence: "Data confidence is X because Y")

*Source: Generated narrative from gantt-release-monitor or manual*

### Section 4: Feature Delivery

Per-family breakdown. This is NOT the bug section.

For each family (CN5000, CN6000, Shared):

## 9. Operating Rules

### Feature rules

- Feature tickets belong to one target release
- Every story/task belongs to an epic
- Every epic belongs to one family delivery root
- Feature progress is measured by non-bug issue completion only

### Bug rules

- Bugs are tracked separately from feature completion
- Bugs may carry multiple fixVersions only for intentional backports
- Bugs link to blocked feature work via `blocks` links when relevant
- Bugs do NOT automatically define feature progress

### Reporting rules

- Feature views use non-bug filters only
- Bug views use bug filters only
- Shared/cross-family work is shown separately, not silently duplicated in totals
- Executive summaries must state whether showing snapshot or live data
- The Data Confidence section is a standing cleanup queue, not decoration

### What automation should NOT do

- Unilateral Jira restructuring without human review
- Closing or moving tickets without owner approval
- Changing fixVersions or parents without PM signoff
